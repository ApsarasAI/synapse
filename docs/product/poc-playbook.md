# PoC Playbook

## 1. PoC 目标

Synapse v1 PoC 不是泛化评估，而是验证一个具体 agent 场景能否在客户环境中完成受控执行。

推荐默认场景：

- PR Review Agent

## 2. PoC 前提条件

客户进入 PoC 前应满足：

- 明确一个真实业务场景
- 有可用的 Linux 部署环境
- 可提供一个最小接入 owner
- 已确认数据边界与部署方式

## 3. 推荐流程

### 阶段一：方案确认

- 对齐业务场景
- 对齐输入输出边界
- 对齐成功标准

### 阶段二：环境部署

- 完成运行环境准备
- 完成 runtime 导入与验证
- 完成 `/health`、`/execute`、`/metrics` 验证

### 阶段三：集成联调

- 用官方 Python SDK 接入
- 跑通标准 demo
- 接入客户真实 prompt / workflow

### 阶段四：验收复盘

- 记录成功指标
- 记录失败与阻塞
- 判断是否进入设计合作阶段

## 4. 成功标准

- 在客户环境稳定执行至少一个真实场景
- 审计与错误语义满足客户评估要求
- 客户能在不依赖研发手工救火的情况下复现 demo 或主路径

## 5. 常见失败原因

- 宿主缺少 Linux 依赖
- Python runtime 未正确导入
- 客户目标场景超出 v1 边界
- 安全预期与当前网络策略能力不一致

## 6. 输出物

每次 PoC 结束后至少产出：

- 场景描述
- 接入路径
- 成功指标结果
- objections 列表
- 下一步建议

推荐统一使用：

- `docs/product/objections-log-template.md`
